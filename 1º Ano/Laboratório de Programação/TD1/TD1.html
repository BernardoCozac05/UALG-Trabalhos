Compilação separada, gestão da compilação e modo Debug

Quando programamos "à séria", muito raramente o nosso projecto de software é escrito num só ficheiro fonte. Organizamos o nosso código conforme a sua estrutura. Por exemplo a interface gráfica vai num ficheiro separado, a interface com uma base de dados noutro, etc.

É mais fácil organizar o código onde cada ficheiro representa uma componente lógica do projecto e a manutenção do código é facilitada (debug incluido) etc...

Como compilar um programa distribuído por vários ficheiros?

Temos de compilar cada ficheiro separadamente conforme a ordem de dependência:

Se o conteúdo do ficheiro f1.c for utilizado no ficheiro f2.c, então f2 depende de f1 e temos então de compilar f1.c antes de f2.c.

Imaginemos que juntamos a estes dois ficheiros o ficheiro main.c que contém a função main. Assim o projecto considerado tem três ficheiros dados na ordem de dependência: A compilação pode fazer-se desta forma:

xxxxxxxxxx
gcc -c f1.c
gcc -c f2.c
gcc -c main.c
gcc f1.o f2.o main.o -o executavel

Primeiro compila-se os ficheiros .c conforme a ordem de dependência, com a opção -c (para "compile only"). Ou seja, compila-se f1.c seguido de f2.c e só depois main.c.

A opção -c serve para indicar ao compilador que não queremos gerar um executável com o ficheiro por compilar, mas sim gerar o ficheiro objecto (de extensão .o). Para gerar o executável precisamos de todos os ficheiros objectos do projecto.

Mas o que é um ficheiro objeto? Um ficheiro objeto é um ficheiro compilado mas sem dependência (tecnicamente dizemos: sem linking, sem ligação).

A título de exemplo, a compilação de um ficheiro .c que usa a função scanf com a opção -c, produz um ficheiro objecto onde aparece claramente a referência de que em certos pontos do seu código é preciso executar função scanf, mas não tem informação de onde está concretamente o código objecto da função scanf no computador em que está a ser executado. O código está no ficheiro objecto da biblioteca stdlib que foi compilada e instalada separadamente. É o que faz a fase de linking: interligar os códigos entre si.

Usemos uma analogia: o planeamento urbano. Compilar ficheiros (para ficheiros objecto) é construir bairros. Fazer a ligação e gerar o executável é, para além da construção dos bairros, construir as infra-estruturas de estradas, de águas e de electricidade que ligam os bairros entre si e os tornam funcionais.

Para facilitar este processo de compilação de projectos de software arbitrariamente complexos, existe um utilitário chamado make. Propomos que se use o make no computador pessoal.

O nosso projecto de software é aqui relativamente simples. Temos dois ficheiros, intdynvec.h e intdynvec.c, e temos o nosso ficheiro principal, o ficheiro main.c que contém a definição a função main. 

xxxxxxxxxx
Dynvec/
│── intdynvec.h  # Ficheiro .hj (API pública)
│── intdynvec.c  # Ficheiro .c, implementação
│── main.c       # Ficheiro principal que testa a biblioteca
│── Makefile     # Ficheiro de configuração do projeto C

 

Usar o utilitário make, desde que está instalado e configurado é muito simples. Em geral basta invocar make (ou de forma equivalente make all).

xxxxxxxxxx
% make
gcc -c intdynvec.h 
gcc -c intdynvec.c
gcc -c main.c
gcc intdynvec.o main.o -o intdynvec

Automaticamente o utilitário make sabe que ficheiros compilar, com que opções e em que ordem. Se eu alterar um destes ficheiros, e invocar make, ele saberá que ficheiros devem ser recompilados e quais não precisam e ser considerados.

Esta magia faz-se porque se deu ao utilitário make um ficheiro de configuração chamado Makefile. Este está na diretoria onde os ficheiros do projecto estão.

Por exemplo, este será o Makefile que vamos usar para a o projeto da primeira parte desta ficha.

xxxxxxxxxx
CC=gcc
CFLAGS=
CCFLAGS=-Wall -Wextra
DYNVEC=intdynvec
MAIN=main
RM=rm -rf
OUT=intdynvec


all: build


build: $(DYNVEC).o $(MAIN).o
  $(CC) $(CCFLAGS) $(CFLAGS) -o $(OUT) $(DYNVEC).o $(MAIN).o


debug: CFLAGS+=-DDEBUG_ON
debug: build


$(MAIN).o: $(MAIN).c $(DYNVEC).h
  $(CC) $(CCFLAGS) $(CFLAGS) -c $(MAIN).c


$(DYNVEC).o: $(DYNVEC).h $(DYNVEC).c
  $(CC) $(CCFLAGS) $(CFLAGS) -c $(DYNVEC).c


clean:
  $(RM) *.o $(OUT) *.gch

De forma resumida, este ficheiro indica que o comando make com a opção all (que é opção por omissão) depende da diretiva build que tem de executar para seguir com o seu rumo.

A diretiva build diz que antes de fazer o seu trabalho, tem de se tratar das diretivas dynvec.o e main.o. Diz-se que a diretiva build depende das diretivas dynvec.o e main.o.

Em seguida, executará o comando

xxxxxxxxxx
gcc -Wall -Wextra -o intdynvec intdynvec.o main.o

Repare no uso de variáveis como CCFLAGS (que vale -Wall -Wextra) ou OUT (que vale intdynvec) para simplificar o Makefile e torná-lo facilmente parametrizável. Se quero mudar o nome do executável para exec por exemplo, qualquer que seja a sua utilização no Makefile, basta-me alterar o seu valor no cabeçalho do ficheiro, ou seja OUT=exec. Isto é semelhante ao #define MAX 100 do C.

Uma das vantagens do uso do utilitário make na compilação e um projeto de software é a gestão automática das dependências. Por outras palavras, se fizemos alterações no ficheiro main.c, precisamos de recompilar o dynvec.c? A partida não. Se alteramos o dynvec.c ou o dynvec.h, então temos de recompilar o main.c porque este depende destes dois ficheiros. O utilitário make calcula as dependências entre ficheiros (com a ajuda do Makefile) e trata automaticamente de compilar somente os ficheiros necessário.

Estruturar o código e os seus serviços na forma de bibliotecas C

Se pretendemos criar um conjuntos de serviços computacionais (por exemplo uma interface para uma base de dados "revolucionária acabada de chegar"), podemos organizar o código de tal forma que outros programadores possam usar estes serviços nos seus programas: as bibliotecas.

Para esse efeito organizamos o código para partilhar em dois ficheiros.

O primeiro ficheiro, designado de ficheiro header de extensão .h.

Neste ficheiro, junta-se estrategicamente a declaração dos serviços por partilhar. Estes serviços podem ser constantes, variáveis, tipos de dados ou funções. São estes serviços que serão disponibilizados aos utilizadores da biblioteca.

Num segundo ficheiro, .c desta vez, fornece-se as implementações dos serviços declarados.

Um exemplo simples é o do exercício anterior. Ali definimos um serviço trivial de soma de inteiros que chamamos add. Para todos os efeitos práticos, basta usar o ficheiro .h para poder aceder aos serviços declarados.

Este facto confere a este esquema uma modularidade interessante: o programador pode usar os serviços sem sequer se preocupar como estão de facto implementados.

O autor pode alterar o código para uma versão mais eficiente, sem nenhum transtorno nem mudança para quem os utiliza, desde que o .h não mude.

Para utilizar estes serviços, basta incluir o .h nos ficheiros do projecto:

xxxxxxxxxx
#include "utils.h"
...

Como se vê, este mecanismo simples:

  1. Facilita a reutilização de código: Escreve-se uma vez e utiliza-se em todos os .c que o programador achar pertinente.

  2. Separa a interface dos serviços da implementação: a interface é o contrato público dos serviços que oferecemos. Como o fazemos é acessório para o utilizador. Desde que o contrato - a interface-- se mantenha, podemos trocar a implementação quando conveniente, e até ter várias implementações alternativas que podem concorrer entre si. A título de ilustração, para uma mesma biblioteca contratualizada pelo ficheiro header .h, podemos ter uma implementação dedicada para o Raspberry Pi, outra dedicada ao Arduino, uma super otimizada, outra mais poupada nos seus gastos energétios etc. É um ponto muito importante! Esta característica participa de um dos mecanismos de base das linguagens de programações poderosas: a modularidade. Podemos também controlar o que fica público e o que fica privado, isto é, o que é necessário mas que não se pretende mostrar ao cliente da biblioteca.

  3. Facilita a fatorização o código e evita duplicações de código. Se vários ficheiros recorrem sistematicamente ao mesmo código, então justifica-se a criação de uma biblioteca onde este código é estruturado em .h e .c.

Um detalhe por tratar e a sua solução

Imagine a situação seguinte:

  • Um serviço apresentado num ficheiro service.h.

  • Um ficheiro f1.c (com o seu header f1.h) que precisa deste serviço.

  • Um ficheiro f2.c que precisa deste serviço mas de f1.c também. 

xxxxxxxxxx
// f1.h + f1.c
#include "service.h"
...
xxxxxxxxxx
// f2.c
#include "service.h"
#include "f1.h"
...

Questão: o ficheiro service.h não será ele carregado duas vezes no f2.c (directamente pelo include e via o include do f1.h) ?

A resposta é sim.

Como evitar esse problema?

Usar uma guarda de inclusão. Ou seja: definir o ficheiro header da biblioteca da seguinte forma:

xxxxxxxxxx
#ifndef BIB_H //BIB_H ou qualquer outro nome
#define BIB_H


// código do header da biblioteca
...
  
#endif // BIB_H

Explicação: Este código, que assume uma intervenção salutar do pre-processador de C, funciona da seguinte forma: primeiro verificar se a guarda BIB_H não está definida (via #ifndef). Se não está definida, isso significa que a biblioteca não foi ainda carregada.

Neste caso, a compilação considera todo o código presente até a diretiva #endif.

A linha a seguir define precisamente a guarda BIB_H. Qualquer outro #include "service.h" terá um #ifndef BIB_H que falhará e logo devolverá um ficheiro vazio.

Gestão dos erros e modo Debug

O ultimo elemento de gestão de projectos de software C que veremos nesta ficha é o de uma gestão cómoda dos erros e de um modo debug.

Primeiro ingrediente

Como facilmente ativar ou desativar um trecho de código dentro de um projeto de software escrito em C? Para quê?

Comecemos pela segunda parte. Idealmente pretendemos dispor de um programa para o qual seja fácil ativar ou desativar a impressão de informação de debug . A vantagem é poder controlar este mecanismo via as opções de compilação e não ter de mudar o código fonte.

Por exemplo, o trecho de código

xxxxxxxxxx
...
#ifdef DEBUG_ON
    x = x + 1;
    printf("DEBUG x=%d\n", x);
#endif
...

com a compilação

xxxxxxxxxx
gcc -DDEBUG_ON ficheiro.c

vai incluir as duas linhas no código compilado, porque definimos a guarda DEBUG_ON na linha de comando da compilação do ficheiro ficheiro.c.

Com a compilação 

xxxxxxxxxx
gcc ficheiro.c

estas duas linhas serão ignoradas.

Exercício

Proponha um novo serviço aritmético de divisão de inteiros na forma de dois ficheiros div.h e div.c . Quando compilado com a guarda DEBUG_ON a função div da biblioteca mostra uma mensagem que imprime o valor dos seus parâmetros a cada invocação para além de efetuar a divisão.

Por exemplo

xxxxxxxxxx
...

deverá imprimir

xxxxxxxxxx

Para além de atribuir a x o resultado da divisão, como esperado.

Se a guarda DEBUG_ON não for definida, a função não imprime a mensagem.

Segundo ingrediente

O problema que pretendemos aqui resolver é o seguinte:

Quando pretendemos definir uma função de protótipo

xxxxxxxxxx
int f(int x);

Como lidar com situações em que a função encontra um erro e não pode devolver o resultado pretendido? Pode sempre mostrar uma mensagem de erro, certo, mas como pode informar as funções que a chamaram?

Devolve o valor -1 para assinalar um erro?

Não é boa ideia. Pois, o valor -1 pode ser um valor válido por devolver.

A linguagem C não tem tipos opção (como nalgumas linguagens funcionais como OCaml ou Haskell), não tem excepções (como Java, C#, OCaml, etc.).

As soluções em C são várias, mas vamos aqui descrever uma em particular: a biblioteca errno dada pelo header errno.h.

Esta biblioteca fornece uma variável global errno , umas constantes e umas funções relacionadas.

A variável errno é a variável que contém valores de erro que as funções vão atualizando.

Se uma função f deteta um erro, esta pode atribuir a errno um valor inteiro convencionado. A função que chamou f pode verificar qual o valor de errno após a chamada para descobrir se f assinalou um erro via errno.

Este é o princípio de base.

Colocamos aqui alguns dos muitos erros que se pode assinalar com errno.

Há também algumas funções auxiliares, como perror(mensagem) e strerror(errno).

  • strerror(errno) produz uma mensagem associada ao valor de errno.

    xxxxxxxxxx
    errno = ENOENT;
    printf("Error: %s\n", strerror(errno));
    // escreve: Error: No such file or directory

  • perror(texto) imprime o texto seguido de uma mensagem associada ao valor de errno.

    xxxxxxxxxx
    FILE *file = fopen("missing.txt", "r");
    if (!file) {
      perror("File open failed");
    }
    // escreve: File open failed: No such file or directory

    Como vê, a função fopen utiliza errno, que inicializou ao valor ENOENT quando detetou o erro e devolveu o valor NULL. Assim poderíamos também ter detetado o erro com if (errno==ENOENT)...

Um exemplo simples:

xxxxxxxxxx
...
errno = 0; // na verdade, esta linha não é necessária
double result = exp(1000);  // too big!


if (errno == ERANGE)  perror("Erro: Overflow matemático");
else                  printf("Resultado: %f\n", result);
// produz: Erro: Overflow matemático: Numerical result out of range
Exercício

Estenda o exercício anterior (o serviço de divisão de inteiros) com este mecanismo de deteção e publicação de erro.

Quando uma divisão por zero é detetada, a função deve inicializar a variável errno com o erro adequado (EDOM). Com o recurso a guarda de DEBUG_ON complementar esta precaução e apresentar uma mensagem de erro adequada dentro da biblioteca.

Para o efeito, considere o seguinte ficheiro main.c

xxxxxxxxxx
#include <stdio.h>
#include <errno.h>
#include <string.h>  // Para strerror()
#include "div.h"


int main() {
    int a = 10, b = 2;
    #ifdef DEBUG_ON
    printf("[DEBUG] Testar div!\n");
    #endif
    // Tentativa de divisão normal
    int result = div(a, b);
    if (errno == EDOM)
         printf("Erro: Divisão por zero! (%s)\n", strerror(errno));
    else printf("%d / %d = %d\n", a, b, result);


    // Tentativa de divisão por zero
    b = 0;
    result = div(a, b);
    if (errno == EDOM)
         printf("Erro: Divisão por zero! (%s)\n", strerror(errno));
    else printf("%d / %d = %d\n", a, b, result);


    return 0;
}

 

 

 

Implementação de uma biblioteca para os vetores dinâmicos

Chegamos agora ao problema central desta ficha.

Primeiro, vamos estudar uma nova estrutura de dados sem nos preocupar com a parte da modularidade própria a definição de uma biblioteca genérica.

Queremos definir um tipo particular de vector em C: os que podem alterar o seu tamanho conforme as necessidades.

Para simplificar esta exposição inicial, vamos falar só de vetores de inteiros.

O princípio de base é o seguinte:

  • se o vetor ficar cheio, então o vetor redimensiona-se para juntar espaço disponível;

  • se, ao contrário, conter pouco elementos, então descarta o espaço disponível para evitar desperdício de espaço memória.

Como vemos, temos um vetor com um tamanho total que se pretende variável. Percebemos assim que temos de saber a qualquer momento quantos elementos estão realmente arquivados no vetor.

Tais vetores são designados de ArrayList em Java ou de vector em C++. Chamemo-lhes de dynvec (para vetores dinâmicos) neste exercício.

Vamos recorrer à palavra chave typedef para introduzir uma declaração de tipo. Relembramos que o uso geral de typedef é:

xxxxxxxxxx
typedef definição_do_tipo nome;

No vetor queremos manter três informações co-relacionadas: os dados arquivados, quantos são, e a capacidade atual do vetor. Para tal usamos uma estrutura, struct em C.

Relembramos que o uso geral de struct é:

xxxxxxxxxx
struct nome_opcional_mas_util {
  tipo_campo_1 nomecampo_1;
  ...
  tipo_campo_n nomecampo_n;
}

Assim, a nossa primeira proposta de definição é:

xxxxxxxxxx
typedef struct Dynvec {
  int *data;       /* os dados/inteiros arquivados no vetor */
  size_t capacity; /* capaciade total do vetor  */
                   /* invariante: capacity >= 0 */
  size_t length;   /* quantidade atual de elementos arquivados no vetor, com o invariante: 0<=length<=capacity */
} dynvec;

  • data é o campo do vetor propriamente dito;

  • length é o campo que informa quantos elementos estão atualmente arquivados no vetor;

  • capacity é o campo que informa da capacidade (máxima) atual do vetor

Repare que

Elementos de Algoritmos e de Estrutura de Dados

Um elemento desta estrutura, ao contrário dos vetores nativos de C, sabe sempre as suas dimensões, tanto físicas (o campo capacity) como lógicas (o campo length). É com base nestas duas informações que podemos estimar se há muito desperdício de memória (espaço livre no vetor data que, concretamente, não é usada, neste caso capacity é bem maior do que length), ou ao contrário constatar que o vetor está no limite da sua capacidade (capacity está apenas maior ou até igual a length).

Juntar ou retirar um elemento ao vetor altera o valor do campo length. Concretamente, incrementa-se ou decrementa-se este campo.

Duas situações merecem particularmente a nossa atenção:

  • quando o vetor está cheio (capacity=length), já não há possibilidade física para juntar novos elementos. Escolhemos aumentar a capacidade real do vetor para arranjar espaço para futuros usos. Isso obriga a realocar espaço memória para o campo data e copiar neste novo espaço o conteúdo atual do vetor. É uma operação potencialmente custosa.

    A questão que se coloca neste caso é: que tamanho físico escolher para o novo vetor? Mais uma célula, mais 10 células? Quantas?

    Os estudos feitos mostram que, em média, a melhor escolha é duplicar o tamanho.

  • quando se pode considerar que o vetor está demasiado vazio? Estudos mostram que o convém considerar demasiado como sendo 3/4 vazio (ou, de forma equivalente, 1/4 ocupado). Estes mesmos estudos indicam que em média, a redução ótima é a de metade do seu comprimento atual. De fato, tal redução permite assegurar que o vetor possa voltar crescer para o dobro do seu tamanho real sem necessitar realocação imediata. E, em média, como os estudos mostram, este palpite (em informática dizemos heurística no lugar de palpite) funciona bem!

Implementação da biblioteca

Para ilustrar como pretendemos construir a biblioteca, vamos dar o código de uma das funções e explica-lo passo a passo. As restantes serão alvo dos exercícios.

O serviço em causa é o serviço de criação de um vetor dinâmico. O seu protótipo é:

xxxxxxxxxx


dynvec *dynvec_create  (void);

Esta função cria a versão inicial de um vetor dinâmico de inteiros, inicializando cada um dos seus campos. A função devolve uma apontador para esta estrutura recém criada.

Uma decisão no desenho da biblioteca é o de escolher que o tamanho inicial de um vetor dinâmico é 4. É uma heurística simples e razoável. Na sua versão inicial o vetor pode albergar até 4 valores sem necessitar de uma alteração de tamanho imediata.

Para tal definimos uma macro em intdynvec.c que fixa este valor

xxxxxxxxxx
#define DYNVEC_INIT_CAPACITY 4

Para criar dinamicamente um valor de tipo dynvec temos de alocar memória para tal. Como sabemos, é o propósito da função malloc.

A função malloc precisa do tamanho do bloco de memória por criar. Este tamanho pode ser dado pelo valor de sizeof(dynvec) .

xxxxxxxxxx
dynvec *v= malloc(sizeof(dynvec));

Se a função malloc for incapaz de encontrar um bloco de memória adequado, esta irá devolver o apontador NULL. A função dynvec_create tem então de reagir defensivamente neste cenário.

Como queremos usar os mecanismos da gestão de erros fornecida pela biblioteca errno.h em conjunto com o modo DEBUG_ON, este mecanismo de defesa é dado pelo código:

xxxxxxxxxx
if (!v){          // se o apontar for v for NULL
  errno = ENOMEM; // assinalar o erro com o código adequado
#ifdef DEBUG_ON   // Em caso de modo DEBUG_ON, 
                  // juntar o código presente até a
                  // directiva #endif
  
                  // mostrar uma mensagem de erro com perror
  perror("Create. Erro ao alocar memória");
#endif
  return NULL;    // e finalmente, devolver o valor NULL 
}

Tomadas estas precauções, estamos em medida de inicializar concretamente o vetor dinâmico.

Para esse efeito, relembramos que o campo X de uma estrutura apontada por um ponteiro p é referenciado por *(p).X. A linguagem C propõe o açúcar sintático seguinte para esta construção: p->X. 

xxxxxxxxxx
// definir a capacidade inicial 
v->capacity = DYNVEC_INIT_CAPACITY;


// no início, não há nenhum elemento no vetor  
v->length = 0;


// aloca-se dinamicamente espaço para o vetor, 
// com v->capacity células de inteiros 
v->data = (int *)malloc(sizeof(int) * v->capacity);

Notemos que o vetor tem inicialmente 4 células de tipo inteiro. O tamanho deste espaço é comodamente dado por sizeof(int) * v->capacity.

Finalmente é preciso tratar do caso em que a função malloc assinala a impossibilidade de alocação de memória, devolvendo o apontador NULL.

Repetimos para esse efeito os cuidados tomados com a alocação de v e terminamos neste caso a devolução do valor NULL.

Finalmente, passado todas estas precauções, estamos em medida de devolver o apontador para o vetor dinâmico devidamente inicializado.

Como um todo, o código da função é:

xxxxxxxxxx
// criar um dynvec com a sua configuração inicial mínima
dynvec *dynvec_create(void){
  dynvec *v= (dynvec *)malloc(sizeof(dynvec));
  if (!v){
    errno = ENOMEM;
#ifdef DEBUG_ON
    perror("Create. Erro ao alocar memória");
#endif
    return NULL;
  }
  v->capacity = DYNVEC_INIT_CAPACITY;
  v->length = 0;
  v->data = (int *)malloc(sizeof(int) * v->capacity);
  if (!(v->data)) {
    errno = ENOMEM;
#ifdef DEBUG_ON
    perror("Create. Erro no array, ao alocar memória");
#endif
    free(v);
    return NULL;
  }
  return v;
}

Vetores dinâmicos genéricos

É possível enriquecer a biblioteca intdynvec com mais alguns serviços interessantes. Mas vamos aqui olhar para uma limitação importante: temos vetores dinâmicos mas só para inteiros.

Como fazer se quiséssemos um vetor dinâmico de valores flutuantes? De cadeias de caracteres?

Esta biblioteca de nada nos servirá nestas situações.

Podemos fazer um clone desta biblioteca para cada uma das situações: floatdynvec.h (e .c), stringdynvec.h (e .c).

Funcionará, mas não é satisfatório, porque obviamente cada nova necessidade obriga a mais uma duplicação de código. O bom programador incomoda-se com código duplicado.

Imagine só descobrir um erro no intdynvec.c. Precisaríamos de corrigir este erro inúmeras vezes, pelos clones todos e nem sempre sabemos quem fez e tem controlo sobre clones desta biblioteca.

As linguagens de programação sérias devem propor mecanismos de modularidade e de programação genérica. A linguagem C propõe tais mecanismos, mesmo se não são dos mais modernos ou práticos. Mas funciona.

Como? Conhecer o tamanho do tipo de dado, usar void, apontadores para void, e o mecanismo de cast (conversão de tipos).

Concretamente no caso dos vetores dinâmicos, isso significa que o campo data será um vetor que vai arquivar dados de um tamanho arbitrário. Se tivermos 10 células no vetor, todas estas células terão o mesmo tamanho, o tamanho do tal tipo de dado. E o vetor terá como tamanho real o tamanho deste tipo de dado vezes 10.

Como não sabemos de antemão na criação da biblioteca de que tipo de dado será o conteúdo do campo data, este terá de ser declarado como void *.

A outra preocupação é como aceder a determinadas posições no vetor? Já abordamos parte da solução a este problema. Lembre-se.

xxxxxxxxxx
t[i]  <--- é açucar sintácico para ---> *(t + i)

A construção *(t + i) é uma construção rica em conhecimento. Assenta numa característica poderosa de C (mais, lembra-se, com poder vem responsabilidades): a aritmética de apontadores.

Em t + i, t é um apontador e i é um inteiro (ou, um size_t). Esta soma é compilada como representando um apontador para uma zona memória que está i blocos de memória a seguir a t. O compilador tem de ser absolutamente preciso sobre o valor do apontador. Se t é, imaginemos, o endereço 12679132 o que será concretamente (t + i)? 12679132 + i? Não.

Primeiro, lembre-se que em C, a unidade de informação memória é um byte. Se i for 3 e se t apontar para um inteiro, 12679132 + 3 é o endereço onde está o quarto primeiro byte a seguir ao inteiro que ali está arquivado (porque em 0 está o primeiro byte deste inteiro). Não é bem o que entendemos por t[i]. Pior ainda, tomado literalmente 12679132 + 3 é o endereço que está a apontar para o ultimo terço do primeiro inteiro, porque um inteiro é representado por 4 bytes.

Nada de pânico. O compilador sabe desta situação e faz algo bastante mais esperto. Percebe que t é um apontador na memória para um determinado tipo de valor, digamos TIPO para exemplificar. Então t + i é para ser entendido como a posição memória que está a i blocos de memória de t, sendo que estes blocos de memória têm exactamente o tamanho dos valores de tipo TIPO.

Concretamente o que acontece é: t+i ---> t + sizeof(TIPO) * i

Assim, se t= 12679132 , i = 2 e t apontar para um short (t ser de tipo short *), t + i é o endereço 12679132+2×2, ou seja 12679136, ou seja um apontador para segundo short que está a seguir ao short apontado por t. Assim, *(t+i) é o dito short para o qual t+i aponta.

 

Parece complicado, mas note que tudo é feito pelo compilador que percebe de que tipo de informação se trata e faz as contas de acordo.

Voltando ao nosso problema, se queremos ter vetores genéricos, ent~so teremos de usar o tipo void como tipo declarado para os valores contidos no vetor. Isso porque não sabemos o que ali será colocado.

Mas neste caso, o compilador já não consegue sabe ao certo qual é o tamanho dos blocos de memória associado ao vetor e como fazer aritmética de apontadores. Não sabe mais aceder directamente aos elementos do vetor! O programador tem de ajudar o compilador!

Vejamos como fazer, com um exemplo:

Como se vê, é possível, mas é preciso dar mais informação de tipagem explícita ao compilador e em consequência a notação fica mais pesada.

Mas é o que nos permite fazer uma biblioteca genérica de vetores dinâmicos, para qualquer tipo.

Um exemplo

Considere uma função muito simples sobre inteiros: a função swap.

xxxxxxxxxx
void swap(int *a, int *b){
int temp=*a;
  *a=*b;
  *b=temp;
  return ;
}

A execução de swap(&i,&j) irá trocar os valores das variáveis de i e j.

Como fazer de swap uma função genérica capaz de trocar qualquer par de variáveis de qualquer tipo?

O princípio por seguir é considerar que vamos trabalhar os valores sobre os quais queremos ser genéricos a partir dos endereços em que estão, e que estes apontadores são apontadores para void.

xxxxxxxxxx
#include <stdio.h>
#include <string.h>// <---onde está definido a função memcpy
void swap(void *a, void *b, size_t size){
  unsigned char buffer[size]; // um bloco de [size] bytes
  memcpy(temp, a, size);      // copiar size bytes do endereço a para o endereço temp
  memcpy(a, b, size);         // copiar size bytes do endereço b para o endereço a
  memcpy(b, temp, size);      // copiar size bytes do endereço temp para o endereço b
  return ;
}

Para ser genérico, temos de considerar que estamos a copiar uma certa quantidade de informação binária de um sítio para outro. Esta informação binária é organizada blocos de bytes.

Assim, para assumir o papel da zona temporária de arquivo necessária à troca, usamos um vetor de bytes de tamanho adequado. A forma mais simples de o fazer é o de definir um vetor de tipo char ou unsigned char (a norma C recomenda da particularmente este último tipo), porque o tamanho deste tipo, o menor dos tipos de base da linguagens C, é precisamente 1 byte.

O tamanho deste vetor depende do tipo de informação que queremos trocar. Se pretendemos trocar inteiros, precisamos de 4 bytes, se for flutuantes de dupla precisão (double), precisamos de 8 bytes. A solução óbvia é colocar esta informação em parâmetro e passa-la a cada utilização da função swap. É o papel do parâmetro size.

Adaptação do tipo de dados.

Vamos então à generalização dos nossos vetores dinâmicos.

xxxxxxxxxx
 typedef struct {
   void *data;        // apontador para a sequencia dos elementos
   size_t elem_size;  // tamanho dos elementos de datat (em bytes)
   size_t capacity;   // capacidade total da sequência data
   size_t length;     // quantidade atual dos elementos arquivados
} dynvec;

Reparemos que o campo data é agora um vetor de elementos de tipo void. Por isso é um apontador para void.

Estamos aqui a afirmar que vamos lidar com a informação por guardar como sendo grupos de bytes sem realmente nos preocuparmos com o que estes bytes significam.

Serão inteiros? serão string? Serão apontadores, ou até mesmos estruturas? Pouco nos importa.

Mas então, para sermos capazes de os processar adequadamente, temos de ter a informação do tamanho. É o papel do novo campo elem_size. Assim elem_size será 4 se o dynvec alberga inteiros.

xxxxxxxxxx
// criar um dynvec com a sua configuração inicial mínima
dynvec *dynvec_create(size_t elem_size){
  dynvec *v= (dynvec *) malloc(sizeof(dynvec));
  if (!v){
    errno = ENOMEM;
#ifdef DEBUG_ON
    perror("Create. Erro ao alocar memória");
#endif
    return NULL;
  }
  v->capacity = DYNVEC_INIT_CAPACITY;
  v->length = 0;
  v->elem_size = elem_size;
  v->data = malloc(v->elem_size * v->capacity);
  if (!(v->data)) {
    errno = ENOMEM;
#ifdef DEBUG_ON
    perror("Create. Erro no array, ao alocar memória");
#endif
    free(v);
    return NULL;
  }
  return v;
}

De notar que poderíamos em alternativa a void * data ter declarado o campo data como sendo do tipo unsigned char * ou até mesmo char *. Como já foi referido, este nos permitiria fazer aritmética de apontadores de forma mais natural na implementação da biblioteca, mas há também alguns inconvenientes (em comparação ao void *). Preferimos a escolha da declaração como void *.

Para terminar a exposição sobre a passagem à uma biblioteca genérica, falemos de como podemos aceder a elementos e como podemos alterar elementos do campo data.

Acesso a elementos do campo data

Imaginemos que queiramos aceder ao elemento i do campo data. Como não sabemos o tipo de dados que estão no campo data, o recurso a data[i] não funciona, nem *(data + i). Temos de fazer o trabalho que habitualmente o compilador faz por nós e calcularmos explicitamente o endereço onde a informação está.

Concretamente, fazemos

xxxxxxxxxx
(char *)v->data + i * v->elem_size; // significa: apontador para a i-ésima célula de v->data

Notemos o detalhe técnico absolutamente necessário da conversão de tipo do apontador para char *. A linguagem C não permite aritmética de apontadores void *. Temos de localmente fazer o cast para char * para que o compilador feche os olhos. É uma ilustração da pequena inconveniência da escolha de void * que referimos acima. É um truque... mas funciona bem e é uma solução standard absolutamente recorrente na programação C. Ou seja, não é bonito... mas é o que todos usam.

Última nota: quando queremos aceder a um elemento arquivado, a biblioteca não sabe qual é o tipo da informação ali arquivada, logo resta-lhe devolver o endereço. O cliente da biblioteca fica com a responsabilidade de dereferenciar o apontador e assumir o tipo real do objecto que ali está. Para melhor explicar, usemos um exemplo.

Assumimos que a função dynvec_get está implementada e devolve o referido apontador.

xxxxxxxxxx
int *x;
int y;
... // processamento adequado do dynvec v
x = (int *)dynvec_get(v,9); //endereço do décimo elemento de v 
y = *x; //

Alteração de elementos do campo data

A atribuição de um valor num determinado índice de data segue o padrão acima descrito. Temos de calcular explicitamente o endereço do local de alteração. Como não sabemos com que tipo de informação estamos a lidar, a cópia tem de ser directa: copiamos uma séria de bits para uma localização do tamanho que esperamos certo. Este tamanho é-nos dado pelo campo elem_size. Esta cópia "bruta" e baixo nível é-nos fornecido pela função memcpy da biblioteca string.h.

A função memcpy é usada para copiar um bloco de memória de um local para outro. O seu protótipo é:

xxxxxxxxxx
#include <string.h>


void *memcpy(void *dest, const void *src, size_t num);

  • dest é um apontador para a área de destino (onde os dados serão copiados).

  • src é um apontador para a área de origem (de onde os dados vêm).

  • num é a quantidade de bytes por copiar.

A função devolve o apontador dest.

Assim, para os nossos fim, a cópia de um valor numa célula de data pode ser feita da forma seguinte:

xxxxxxxxxx
    memcpy((char *)v->data + arr->length * v->elem_size, element, v->elem_size);

Note que

  • se usou novamente o truque (char *) para poder fazer aritmética de apontadores;

  • o valor por copiar é dado pelo seu endereço, ou seja, passado por referência.

A estrutura de dados Pilha

Organizar serviços em bibliotecas reutilizáveis e programar cuidadosamente de forma a separar as implementações das interfaces é uma hábito mais do que saudável. É desejável.

Vejamos aqui uma ilustração flagrante: a biblioteca que fornece o serviço de uma pilha.

Uma pilha é uma estrutura de dados que arquiva valores em sequência e que permite um acesso que segue a política Last-In/First-Out.

Junta-se um elemento (um dossier, na imagem) na parte de cima. O primeiro dossier por tratar é o de cima, logo o último que se lá pôs.
